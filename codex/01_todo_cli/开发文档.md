## 2. 项目概述

### 2.1 项目名称

`todo-cli`

### 2.2 项目目标

开发一个基于 Node.js 的命令行待办事项管理工具，支持用户通过命令行完成任务的新增、查看、完成和删除操作，并将数据持久化到本地。

### 2.3 项目特点

- 轻量级
- 本地运行
- 无需数据库
- 易于扩展
- 适合个人使用或作为 CLI 项目练习样例

---

## 3. 需求范围

### 3.1 功能范围

首版需支持以下命令：

- `add`：新增待办事项
- `list`：查看待办事项列表
- `done`：将某条任务标记为已完成
- `remove`：删除某条任务

### 3.2 技术要求

- 使用 Node.js 开发
- 数据采用本地存储方式
- 通过命令行方式执行

### 3.3 非功能范围

首版不包含以下内容：

- Web 页面
- 用户系统
- 云端同步
- 数据库接入
- 多端协同
- 复杂筛选和排序功能

---

## 4. 技术方案

### 4.1 开发语言与运行环境

- 语言：JavaScript
- 运行环境：Node.js
- 运行方式：CLI 命令执行

### 4.2 存储方案

采用本地 JSON 文件保存任务数据。

**选择原因：**

- 实现简单
- 便于调试
- 满足首版需求
- 无需引入数据库依赖

### 4.3 架构思路

项目采用轻量分层结构：

- CLI 入口层
- 命令处理层
- 业务逻辑层
- 数据存储层
- 工具辅助层

这种结构适合小型命令行项目，既避免过度设计，也能保证后续维护与扩展。

## 6. 模块划分

### 6.1 CLI 入口模块

**职责：**

- 接收命令行输入
- 解析命令和参数
- 路由到对应功能模块
- 输出执行结果

**输入：**

- 命令名称
- 命令参数

**输出：**

- 成功提示
- 错误提示
- 列表信息

### 6.2 命令处理模块

包含以下子模块：

- `add`
- `list`
- `done`
- `remove`

**职责：**

- 接收解析后的参数
- 调用业务服务
- 返回统一格式输出

**设计原则：**

- 每个命令独立
- 不直接处理底层存储
- 不承担复杂业务判断

### 6.3 任务业务模块

**职责：**

- 封装任务新增逻辑
- 封装任务查询逻辑
- 封装任务状态变更逻辑
- 封装任务删除逻辑

**说明：**

业务模块是系统核心，应集中处理规则，避免规则分散在命令层和存储层中。

### 6.4 本地存储模块

**职责：**

- 读取任务列表
- 写入任务列表
- 初始化存储文件
- 处理文件异常

**说明：**

存储模块只负责文件层面的读写，不参与业务判断。

### 6.5 数据模型模块

**职责：**

- 统一定义任务的数据结构
- 保证读写数据格式一致

**任务对象建议字段：**

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | string | 任务唯一标识 |
| `content` | string | 任务内容 |
| `status` | string | 任务状态，`pending` 或 `done` |
| `createdAt` | string | 创建时间 |
| `updatedAt` | string | 更新时间 |

### 6.6 工具辅助模块

**职责：**

- 参数校验
- 输出格式化
- 文件路径处理
- 错误信息统一封装

## 7. 功能设计

### 7.1 `add` 功能

**功能说明：**

新增一条待办任务。

**输入：**

- 任务内容

**处理逻辑：**

1. 接收任务内容
2. 校验任务内容是否为空
3. 生成任务对象
4. 写入本地存储
5. 返回新增结果

**输出：**

- 新增成功提示
- 参数错误提示

**异常场景：**

- 内容为空
- 数据文件无法写入

### 7.2 `list` 功能

**功能说明：**

展示当前全部任务。

**输入：**

- 无必填参数

**处理逻辑：**

1. 读取本地任务列表
2. 按统一格式输出任务
3. 处理空列表场景

**输出：**

- 当前任务列表
- 空列表提示

**异常场景：**

- 数据文件不存在
- 数据文件格式异常

### 7.3 `done` 功能

**功能说明：**

将指定任务标记为已完成。

**输入：**

- 任务 ID

**处理逻辑：**

1. 根据 ID 查找任务
2. 判断任务是否存在
3. 更新任务状态为已完成
4. 更新修改时间
5. 写回本地数据

**输出：**

- 标记成功提示
- 未找到任务提示
- 重复完成提示

**异常场景：**

- ID 缺失
- ID 无效
- 数据写入失败

### 7.4 `remove` 功能

**功能说明：**

删除指定任务。

**输入：**

- 任务 ID

**处理逻辑：**

1. 根据 ID 查找任务
2. 判断任务是否存在
3. 删除任务
4. 写回本地数据

**输出：**

- 删除成功提示
- 未找到任务提示

**异常场景：**

- ID 缺失
- ID 无效
- 删除后写入失败

## 8. 数据设计

### 8.1 存储文件

建议使用：

`data/tasks.json`

### 8.2 数据格式示意

```json
[
  {
    "id": "1",
    "content": "学习 Node.js CLI",
    "status": "pending",
    "createdAt": "2026-03-26T10:00:00.000Z",
    "updatedAt": "2026-03-26T10:00:00.000Z"
  }
]
```

### 8.3 数据规则

- `id` 必须唯一
- `content` 不能为空
- `status` 仅允许 `pending` 或 `done`
- 创建任务时写入 `createdAt` 和 `updatedAt`
- 修改任务时更新 `updatedAt`

- 可成功新增任务
- 可查看任务列表
- 可按 ID 标记任务完成

### 12.3 文档验收

- README 包含安装和使用说明

## 13. 风险与注意事项

### 13.1 数据文件损坏风险

**风险：**

本地 JSON 文件被手工修改或写入中断，可能导致格式损坏。

**应对：**

- 增加格式校验
- 读取失败时提供错误提示
- 初始化逻辑保持简单清晰

### 13.2 命令参数不规范风险

**风险：**

用户输入缺失、格式错误或无效 ID。

**应对：**

- 增加统一参数校验
- 提供帮助提示
- 对异常输入做兜底处理

### 13.3 后续扩展耦合风险

**风险：**

若命令逻辑、业务逻辑、存储逻辑混在一起，后续扩展困难。

**应对：**

- 保持模块职责分离
- 统一由服务层处理核心逻辑
- 存储层只负责读写

---

## 14. 后续扩展建议

首版完成后，可按需要扩展：

- `edit`：编辑任务内容
- `clear`：清空全部已完成任务
- `list --done`：只查看已完成任务
- `list --pending`：只查看未完成任务
- `search`：按关键词搜索任务
- 支持自定义存储路径

## 15. 结论

`todo-cli` 项目适合采用轻量级 Node.js CLI 架构实现。

首版重点是以最小可用功能完成任务管理闭环：

- 有命令入口
- 有任务读写
- 有本地持久化
- 有基础异常处理
- 有清晰的模块边界

该方案开发成本低、实现路径清晰，适合快速落地并支持后续迭代。
